# Tutor v3 Implementation Plan

Task description: `docs/tutor-v3-spec.md`

Repo guidance read first: `CLAUDE.md`

This file records the Tutor v3 implementation plan and acceptance criteria for this task.

## 1. Approach

Use an audit-first, plugin-boundary-preserving implementation strategy.

The task is not greenfield. The current checkout already contains substantial Tutor v3 structure under `plugins/tutor/`, with supporting regression tests in `packages/core/test/`. Before changing behavior, reconcile the spec against the code so future work extends the existing implementation instead of duplicating concepts under new names.

Methodology:

1. Re-audit `docs/tutor-v3-spec.md` against `plugins/tutor/`, `packages/shared/`, `packages/core/lib/`, and `packages/core/test/`.
2. Keep tutor-specific behavior in `plugins/tutor/`. The existing monorepo pattern is that plugins import `Plugin`, `ToolDef`, `PluginContext`, `text()`, `err()`, and interaction registries from `@choomfie/shared`; core owns Discord dispatch, MCP routing, config, and plugin loading.
3. Prefer pure helpers and DB-backed state over Discord mocks. Existing tests exercise pure lesson generation, SQLite-backed profile/SRS state, and button option construction directly.
4. Use additive, idempotent SQLite migrations in `LessonDB`, matching the existing `CREATE TABLE IF NOT EXISTS` pattern.
5. Treat Discord interactions as latency-sensitive. Slash commands, buttons, and modals should continue to be registered through shared registries and handled directly without a Claude roundtrip.
6. Avoid widening `PluginContext` unless there is a real cross-plugin API need. For Tutor v3 reminders, plugin-owned reminder state plus direct DM is less coupled than exposing core `ReminderScheduler`.

Implementation order:

1. Lock down existing behavior with tests for catalog shape, tool registration, prompt context, profile persistence, SRS reminder settings, and button custom IDs.
2. Complete or harden Phase A features: chart exercises, random word tool, SRS reminders, furigana levels, and authoring guidance.
3. Complete or harden Phase B features: content-set exercise generation, deterministic learner profile updates, and units 2-4.
4. Only add the B1 user-selected mode picker if lesson scoring is redesigned to score against the active generated exercise set. A picker that still uses `lesson.exercises.length` as the denominator would be incorrect.

## 2. Files To Modify/Create

### Shared package

- `packages/shared/types.ts`
  - Preserve the plugin boundary: `Plugin`, `ToolDef`, `ToolResult`, `text()`, and `err()`.
  - No Tutor v3-specific types should move here unless multiple plugins need them.
- `packages/shared/plugin-context.ts`
  - Keep `PluginContext` minimal. Avoid adding core `ReminderScheduler` for Tutor v3 unless the reminder implementation intentionally becomes a shared plugin API.
- `packages/shared/interactions.ts`
  - Continue using `registerCommand()`, `registerButtonHandler()`, and `registerModalHandler()` for plugin-owned interactions.
  - Do not add dispatch logic here; dispatch belongs in core.

### Core package

- `packages/core/lib/plugins.ts`
  - Preserve explicit workspace package loading through the `PLUGIN_PACKAGES` map.
  - Confirm `tutor: "@choomfie/tutor"` remains registered.
- `packages/core/lib/interactions.ts`
  - Preserve the split architecture: shared registries are re-exported, and core `handleInteraction()` owns dispatch and `safeHandle()`.
  - No tutor-specific branches should be added here.
- `packages/core/lib/discord.ts`
  - Preserve startup ordering: core commands import first, plugin `init()` runs on Discord ready, then `getCommandDefs()` is used for command auto-deploy.
  - Verify plugin command registration is visible to auto-deploy after `@choomfie/tutor` is imported.
- `packages/core/scripts/deploy-commands.ts`
  - Verify whether manual command deployment sees plugin commands. If not, plan a small explicit import/load path rather than relying on side effects accidentally.
- `packages/core/test/lesson-buttons.test.ts`
  - Extend for stable short answer tokens, overlong labels, answer text containing `:`, stale exercise index handling, and rerender stability.
- `packages/core/test/tutor-prompt.test.ts`
  - Extend for full/partial/none furigana context and learner profile injection.
- `packages/core/test/lesson-catalog.test.ts`
  - Maintain catalog invariants for units, duplicate lesson IDs, prerequisites, empty lessons, button distractors, and reachability.
- `packages/core/test/exercise-generator.test.ts`
  - Maintain invariant tests for generated recognition, production, and matching exercises.
- `packages/core/test/learner-profile.test.ts`
  - Maintain deterministic profile update coverage: score averages, strong/weak areas, preferred exercise type, streak updates, and SRS stats.
- `packages/core/test/lesson-db-profile.test.ts`
  - Maintain SQLite round-trip tests for learner profiles and SRS reminder preferences.
- `packages/core/test/srs.test.ts`
  - Extend for `getDueCountByUser()` if reminder behavior changes.
- `packages/core/test/regression/tools-register.test.ts`
  - Keep expected tutor tool names and count aligned with `plugins/tutor/tools/index.ts` and `plugins/tutor/index.ts` `userTools`.

### Tutor plugin

- `plugins/tutor/index.ts`
  - Keep this as the plugin lifecycle owner: initialize `SRSManager`, `LessonDB`, lesson registration, module initialization, reminder timers, and cleanup in `destroy()`.
  - SRS reminders should guard missing `ctx.discord`, swallow DM delivery failures, respect persisted opt-out/cooldown state, and clear timeout/interval handles on shutdown.
  - Keep `userTools` synchronized with exported tools.
- `plugins/tutor/lesson-interactions.ts`
  - Keep `/lesson` and `/progress` registered via `@choomfie/shared`.
  - Keep active lesson session state in memory, keyed by user ID.
  - Use short session-scoped answer option tokens in Discord `customId`s, not raw answer text.
  - For chart and matching exercises, either keep the current one-answer-per-exercise model or add structured multi-step state; do not fake multi-blank charts without session state for the current blank.
  - If B1 user mode selection is implemented, add a mode selection state before the first exercise and store the exact exercise list in `ActiveLessonSession`.
- `plugins/tutor/core/lesson-types.ts`
  - Maintain `Exercise.type` support for `chart` and `matching`.
  - Keep `Lesson.furiganaLevel?: "full" | "partial" | "none"`.
  - If richer chart behavior is implemented, add a structured `ChartExercise` shape rather than encoding grid state only in prompt text.
  - If user-selected mode picker is implemented, move `ExerciseMode`, `ContentItem`, and `ContentSet` here so lesson definitions and generator share one type source.
- `plugins/tutor/core/exercise-generator.ts`
  - Keep content-set-driven generation for recognition, production, and matching.
  - Add pure selection helpers if runtime mode picking is implemented, so tests do not need Discord interaction mocks.
- `plugins/tutor/core/lesson-engine.ts`
  - Preserve mastery-gated progression, prerequisite unlocking, and SRS item import on passed lessons.
  - If active sessions can contain generated subsets, change completion scoring to use that session exercise set, not `lesson.exercises.length`.
- `plugins/tutor/core/lesson-db.ts`
  - Keep lesson progress, learner profiles, and SRS reminder settings in one tutor-owned SQLite DB.

- `plugins/tutor/core/learner-profile.ts`

- `plugins/tutor/core/srs.ts`

- `plugins/tutor/tools/tutor-tools.ts`

- `plugins/tutor/tools/srs-tools.ts`

## 3. Acceptance Criteria

- [x] Phase A1 chart exercises exist for the intended kana review points and render as button exercises without raw answer text in `customId`.
- [x] Phase A2 `random_word` is exported by `@choomfie/tutor`, listed in `userTools`, and returns a random N5 vocab word with hidden meaning.
- [x] Phase A3 SRS reminders check due counts periodically, require at least five due cards, respect a 24-hour cooldown, persist opt-out state, and do not throw when DMs fail.
- [x] Phase A4 `furiganaLevel` is represented in lesson metadata and reaches `tutor_prompt` when the user has an active lesson.
- [x] Phase A5 vocab-first authoring expectations are documented and reflected in unit 3/4 prerequisites.
- [x] Phase B1 content sets can generate recognition, production, and matching exercises, with deterministic testable invariants.
- [x] B1 user mode picker remains out of scope; no generated subset scoring path was added.
- [x] Phase B2 learner profiles update deterministically after lesson completion and SRS review, persist per user/module, and are injected into `tutor_prompt`.
- [x] Phase B3 Japanese units 2-4 are registered, reachable through prerequisites, visible in `/progress`, and exposed through `lesson_status`.
- [x] Tutor plugin lifecycle initializes and destroys DBs, module state, SRS state, and timers cleanly.
- [x] Plugin command registration works through the shared registry and is visible to startup auto-deploy.
- [x] Tool registration tests confirm all tutor tool names and `userTools` are synchronized.
- [x] `bun test` passes.
- [ ] Manual Discord smoke test passes for `/lesson`, button answers, typed answers, lesson completion, SRS import/review, `/progress`, `random_word`, `srs_reminders`, and `tutor_prompt` during an active furigana lesson.

Explicitly out of scope unless separately requested:

- Random-word daily auto-post scheduling.
- LLM-rewritten learner profiles.
- A full multi-blank chart engine if the current one-blank-per-exercise model is accepted.
- User-selected exercise mode picker unless scoring/session state is updated at the same time.

Existing tests to preserve and extend:

  - Covers answer option generation, short custom IDs, label truncation, and stable token-to-answer mapping.
- `packages/core/test/tutor-prompt.test.ts`
  - Covers Japanese prompt shape, furigana directives, active lesson context, and learner profile injection.
- `packages/core/test/lesson-catalog.test.ts`
  - Covers unit registration, duplicate lesson IDs, prerequisite resolution, unit references, exercise validity, and reachability.
- `packages/core/test/exercise-generator.test.ts`
  - Covers recognition, production, matching, empty content sets, and custom mode lists.

  - Covers deterministic learner profile updates from lessons and SRS review, plus prompt formatting.
- `packages/core/test/lesson-db-profile.test.ts`
  - Covers profile round-trips, idempotent DB reopen, reminder opt-out, persisted cooldown, and `srs_reminders`.
- `packages/core/test/srs.test.ts`
  - Covers SRS learned-card counting and card ownership checks; add due-count tests if reminder thresholds change.
- `packages/core/test/plugins.test.ts`
  - Covers core plugin loader behavior for enabled plugins.
- `packages/core/test/regression/plugin-load.test.ts`
  - Covers plugin interface shape.
- `packages/core/test/regression/tools-register.test.ts`
  - Covers plugin tool names and counts.
- `packages/core/test/boot.test.ts` and `packages/core/test/regression/imports.test.ts`

bun test packages/core/test/tutor-prompt.test.ts
bun test packages/core/test/lesson-catalog.test.ts

Avoid:

- Snapshotting randomized output. Assert invariants such as membership, counts, no self-distractors, and required answer inclusion.
- Heavy Discord.js mocks for full interaction flows. Factor button/session logic into pure helpers and test those directly.
- Mocking SQLite for `LessonDB` or `SRSManager`; the repo already uses real temporary DBs.

## 5. Edge Cases & Risks

- Discord `customId` has a 100-character limit and `:` is used as a separator. Store answer text in session memory and put only short tokens in IDs.
- Button labels also have length limits. Truncate labels while preserving full answer text in the session mapping.
- Active lesson sessions are in memory and disappear on worker restart. Expired session paths should tell users to run `/lesson`.
- Typed-answer exercises arrive through `plugin.onMessage` before normal Claude message handling. Ensure tutor consumes only active production/cloze exercises.
- Chart exercises can become misleading if prompt-rendered blanks are not backed by structured state. Either keep one blank per exercise or add proper chart progress state.
- Matching exercises are currently sequential button exercises, not drag-and-drop pair matching. Document this UX choice if accepted.
- Mode picker scoring is the largest correctness risk. The score denominator must match the selected exercise set.
- SRS reminders can spam after restart if cooldown is only in memory. Keep cooldown persisted in `LessonDB`.
- DM reminder delivery may fail when users block DMs or share no mutual server. Treat this as non-fatal.
- `recordSrsReminderSent()` must not accidentally re-enable users who opted out.
- Plugin command deployment depends on side-effect command registration. Manual deploy may need explicit plugin imports if it does not load enabled plugins.
- Lesson catalog randomness can make brittle tests. Keep tests invariant-based.
- Additive migrations must not rewrite existing lesson progress, learner profiles, or SRS cards.
- Shared package changes require at least a worker restart; supervisor/IPC changes require a full Choomfie session restart. Keep scope out of supervisor files.
- SRS imports on lesson completion may duplicate cards if repeated completions are allowed. Decide whether duplicates are acceptable or add idempotency by deck/tag/front.
- The spec mentions using the existing reminder system, but plugins currently depend only on `PluginContext`. Exposing core `ReminderScheduler` would couple plugin code to core internals.
