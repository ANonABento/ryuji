# Security audit (Discord bot — token + permissions + shell-out)

## Scope

Security audit of Choomfie (Discord bot, MCP server, plugin runtime). Discord bots have unique attack surface — focus there.

### 1. Token & secret handling
- `~/.claude/plugins/data/choomfie-inline/token` — confirm correct file perms (0600), not in git, not logged
- Any other secret storage paths
- `.env.example` matches reality, `.gitignore` correct, no leaks in git history

### 2. Permission model — owner vs allowlist vs everyone
- `packages/core/lib/handlers/shared.ts::requireOwner()` — confirm enforcement on EVERY owner-only tool
- `userTools` plugin field — confirm allowlisted plugins can't grant unsafe tool access to non-owners
- Allowlist file (`access.json`) — race condition between in-memory + disk?
- Tool-call permission DM relay — can a non-owner spoof an approval?

### 3. Shell-out audit
- Every `Bun.spawn`, `child_process.exec`, `gh` shell-out, ffmpeg call — confirm:
  - No user-input in command line without sanitization
  - No template-string interpolation of attacker-controllable strings
  - Timeouts everywhere

### 4. MCP tool surface
- Every tool in `packages/core/lib/tools/` AND every plugin tool — confirm:
  - Owner-gating where appropriate
  - Input validation (Zod or equivalent)
  - No SSRF (`browse`, `youtube_*`, `linkedin_*`, etc. all hit URLs — validate)
  - File path traversal (`Read`/`Write`-style tools must not escape `~/.claude/plugins/data/choomfie-inline/`)

### 5. Discord-side risks
- Webhook URLs in config — confirm not logged, not echoed back
- Server vs DM context — rate limits, allowlist applies to DMs too?
- Mentions / replies from untrusted users — can they inject prompt content that escapes context?

### 6. Voice / network
- STT/TTS provider keys (Groq, ElevenLabs) — secret handling
- WebRTC / Discord voice — any unsafe deserialization?

### 7. Dependency audit
- `bun audit` for HIGH/CRITICAL
- Pinned vs floating versions

## Acceptance Criteria
1. ✅ Audit report at `docs/security-audit-2026-05-04.md`: findings, severity, file:line, fix.
2. ✅ All HIGH and CRITICAL fixed in same PR.
3. ✅ MEDIUM fixed or accepted-risk documented.
4. ✅ Test added for every permission-fix (verify a non-owner gets rejected).
5. ✅ Plugin API hardening (#43) was the prior pass — document any gaps still open.

## Test Plan
- Unit: `requireOwner()` and `isAllowed()` exhaustive coverage.
- Integration: mock a non-owner Discord message → assert it gets the right rejection for every owner-only tool.
- Shell-out injection: feed `; rm -rf /` style strings, confirm escapement.
- Path traversal: feed `../../../etc/passwd` to file-path inputs, confirm rejection.
- `bun test` and `bun run type-check` clean.

## Out of scope
- Adding new features
- Refactoring not driven by a finding
- The Tutor plugin's pedagogy (only its data-handling)
- Voice audio quality

Discord bots are nicer attack surfaces than they look — be paranoid about permission checks especially.
