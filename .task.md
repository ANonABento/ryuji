# IMPROVE-07: Profile — completeness score with gap suggestions

Implement spec at `docs/audit/feature-specs/2026-05-10-improve-07-profile-completeness.md`.

Profile page shows raw fields with no signal of "what's good enough". Add a completeness score (0-100) with targeted gap suggestions.

Scoring rubric (100 pts total):
- Contact info (name, email, headline) — 10
- Summary present — 10
- ≥1 experience with ≥3 bullets — 20
- ≥2 quantified bullets across all experiences — 10
- Education present — 10
- ≥5 skills — 10
- ≥1 project (early-career) OR additional senior experience — 10
- Achievements/awards — 10
- LinkedIn/GitHub URL — 10

Display:
- Hero card on profile page: "Profile is 73% complete · 4 quick wins available"
- Per-gap card: "Add 1 quantified bullet to your Anthropic role" → click → focuses input
- Confetti animation crossing 50/75/90/100% thresholds

New/modified files:
- `apps/web/src/lib/profile/completeness.ts` (new) — `scoreProfile(profile) → {score, gaps[]}`
- `apps/web/src/components/profile/completeness-card.tsx` (new) — hero card
- `apps/web/src/components/profile/gap-list.tsx` (new) — actionable suggestion list
- `apps/web/src/app/[locale]/(app)/profile/page.tsx` — mount card at top

Acceptance:
- Card with score visible on profile page
- Gap list shows top 3 actionable items, each click-to-focus
- Score updates live as user fills sections
- Confetti at 50/75/90/100%
- Test fixtures: empty (low score), full (100), edge cases
- type-check + lint clean

Out of scope: AI-generated suggestions, multi-language profiles, public share link, peer comparison.

## 1. Approach

### Methodology

**Refactor + extend the existing `apps/web/src/lib/profile-completeness.ts`, don't replace it.** That file (76 lines, fully tested by `profile-completeness.test.ts` with 13 cases, consumed by `dashboard/page.tsx:134` and `lib/resume/insights.ts:347/370`) already has a working 5-section / 100-pt scorer with `nextAction` and section `href`s. The spec adds 4 new categories, finer-grained gap text, and click-to-focus. We treat the existing module as the v1 and grow it into v2 in place — so dashboard and insights consumers keep working with the same return shape augmented with optional fields.

Three layers:

1. **Pure scoring (`lib/profile/completeness.ts`)** — `scoreProfile(profile): { score, gaps, sections, thresholdsCrossed }`. Pure function, no React, no DOM, fully unit-testable. Re-export `calculateProfileCompleteness` from this new module so existing callers keep compiling, then migrate them in a follow-up.
2. **Presentation (`components/profile/completeness-card.tsx`, `gap-list.tsx`)** — client components that take the score result + an `onFocusField(fieldId)` callback. The card owns the threshold-crossing detection (compares previous score to current via a ref) and triggers the confetti.
3. **Wiring (`app/[locale]/(app)/profile/page.tsx`)** — mount card at top of `PageContent`, derive score from live `form` state (not just saved `profile`) so it updates as the user types. Pass `focusField` (already defined at line 213) as the click-to-focus handler — it already does tab-switch + scrollIntoView + focus, exactly what gap cards need.

### Why this shape

- **Score from `form`, not `profile`.** The existing page already maintains `form: ProfileFormValues` and `profile: Profile | null`. `profile` is the saved snapshot from `/api/profile`; `form` is the live edit buffer. The acceptance criterion *"Score updates live as user fills sections"* requires deriving from `form`. Build a small `formValuesToProfileLike()` adapter (or extend existing `formValuesToProfileUpdate` in `lib/profile-form.ts`) so the same scorer works on both.
- **Confetti as a CSS/SVG burst, no new dependency.** The spec explicitly allows this: *"shadcn or simple css burst"*. `package.json` has no confetti library installed. Adding `canvas-confetti` (~6KB gzipped) is fine if the user wants polish, but a pure-CSS burst keeps the dep tree small. Default to CSS; offer canvas-confetti as a 1-line swap.
- **Pure scorer, separate from the React tree.** Mirrors how `lib/ats/scoring/`, `lib/streak/`, and the existing `profile-completeness.ts` are organised — pure logic in `lib/`, presentation in `components/`. Allows headless reuse from server actions, future CLI exports, or insights pipelines.
- **Threshold tracking via `useRef`-of-previous-score.** Don't store thresholds in localStorage on first pass — the spec doesn't ask for "fire only once per user lifetime", just "crossing thresholds". A ref of the previous score is enough to detect a single edit-session crossing without re-firing on every render. Note the risk in §5.

### Open spec gaps (must resolve before coding two of the rubric items)

The 9-category rubric in the spec doesn't fully line up with `Profile` schema in `packages/shared/src/types.ts`. Two items need a defensible interpretation:

| Rubric item | Schema reality | Proposed interpretation |
|---|---|---|
| "Achievements/awards — 10 pts" | `Profile` has `certifications: Certification[]` but no `awards` field. `Experience.highlights` is the closest thing. | Count as complete if: `certifications.length ≥ 1` **OR** at least one experience/project highlight matches an award keyword regex (`/\b(award|honor|recognition|valedictorian|dean['']s list|patent|grant|scholarship|first place|winner)\b/i`). Document the heuristic in a code comment so reviewers can see why. |
| "≥1 project (early-career) OR additional senior experience (later-career) — 10 pts" | No `careerStage` field. | Heuristic: derive years-of-experience from sum of `(endDate ?? today) - startDate` across `experiences`. `< 4 years` → early-career → require `projects.length ≥ 1`. `≥ 4 years` → senior → require `experiences.length ≥ 2`. The 4-year cutoff is editable as a named constant (`EARLY_CAREER_YEARS`). |
| "≥2 quantified bullets across all experiences" | Highlight is a free-text string. | Quantified = matches `/\d/` AND length > 8 chars (avoids "v2" or "1 yr" being sole qualifier). Apply across `experiences[].highlights[]` and `projects[].highlights[]`. Exact regex tunable. |

These are judgment calls. Flag them in the PR description so the user can dial them.

## 5. Edge cases & risks

### Edge cases

1. **Saved profile already at 100% on page load.** Confetti must not fire on mount. Solved by initialising the prev-score ref to the *initial* score, not `null`. Add an explicit test.
2. **Live score drops** (user deletes their summary). Score going down must not crash the threshold detector. Test: 80 → 50 → 80 fires confetti once (the second 50→80 crossing), or zero times depending on chosen semantics. Pick one, document, test.
3. **Quantified-bullet false positives.** Bullets like "Worked on v2" or "Hired 1 person" are technically quantified but trivially so. Mitigation: require digit + length > 8. Document the heuristic; expose as constant for tuning.
4. **No experiences but has projects (recent grad).** YOE = 0 → early-career bucket → projects requirement applies. Verify bucket selection doesn't crash on empty experiences array.
5. **`endDate` parsing.** `Experience.startDate`/`endDate` are strings (`"2020"` in fixtures, possibly `"2020-03"` or `"2020-03-15"` in real data). Use a defensive parser: `new Date(s)` if invalid → 0 contribution. Don't throw.
6. **i18n.** The page is under `[locale]`. New strings ("Profile is N% complete", "quick wins available", gap CTAs) need to either be hardcoded EN (out-of-scope per spec) **or** routed through the existing translation helper. Check `apps/web/src/app/[locale]/(app)/profile/page.tsx` — the existing strings are hardcoded EN, so matching that is consistent. Note in PR.
7. **Form-vs-profile drift for nested arrays.** `ProfileFormValues` (per `lib/profile-form.ts`) likely doesn't include experiences/education/skills as editable form fields — those are read-only on the profile page (career details rendered at line 519-568). So the live-update only works for fields the form actually owns: name, email, headline, summary, linkedin, github. Experience/skill/education updates only reflect after a save round-trip from a different page (resumes, parsed docs). **This is a real limitation of "live updates"** — the spec's "as user fills sections" language is aspirational for the sections this page doesn't actually edit. Recommend: scope "live" to the editable subset, recompute on `useProfile` SWR/snapshot revalidation for the rest. Confirm with user before implementation.
8. **Confetti accessibility.** Honor `prefers-reduced-motion`: skip the burst, keep the threshold callback. Add a test.
9. **Multiple consumers of `lib/profile-completeness.ts`.** Don't change the v1 API. Adding the v2 alongside keeps `dashboard/page.tsx` and `lib/resume/insights.ts` working. PR should note the v1 will be deprecated in a follow-up.
