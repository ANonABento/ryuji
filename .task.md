# Tutor v3 Implementation Plan

Task source: `docs/tutor-v3-spec.md`

Repository guidance read first: `CLAUDE.md`

Scope note: this plan was written audit-first. The current branch now includes runtime hardening for structured chart completion and user-selected practice modes.

## Current Codebase Audit

The current checkout already contains substantial Tutor v3 work under `plugins/tutor/`:

- A1 chart exercises exist as `Exercise.type === "chart"` plus `chartReview()` and `renderChartGrid()` in `plugins/tutor/modules/japanese/lessons/kana-helpers.ts`.
- A2 `random_word` exists in `plugins/tutor/tools/random-word.ts` and is included by `plugins/tutor/tools/index.ts`.
- A3 SRS due reminders exist in `plugins/tutor/index.ts`, with persisted opt-out/cooldown settings in `LessonDB.srs_reminder_settings`.
- A4 `Lesson.furiganaLevel` exists in `plugins/tutor/core/lesson-types.ts` and is injected by `tutor_prompt` through `plugins/tutor/tools/tutor-tools.ts`.
- A5 lesson authoring guidance exists in `plugins/tutor/LESSON_AUTHORING.md`.
- B1 generated exercise modes exist in `plugins/tutor/core/exercise-generator.ts`, and matching exercises are represented as button exercises.
- B2 deterministic learner profiles exist in `plugins/tutor/core/learner-profile.ts`, persisted by `plugins/tutor/core/lesson-db.ts`, and injected into `tutor_prompt`.
- B3 units 2-4 are registered through `plugins/tutor/modules/japanese/lessons/index.ts`.
- Tests already cover lesson buttons, catalog integrity, exercise generation, learner profiles, SRS, plugin loading, and tutor prompt behavior in `packages/core/test/`.

The richer UX pieces described by the spec, especially sequential multi-blank chart filling and user-selected exercise modes, are no longer cosmetic follow-ups in this branch. They are implemented by changing active lesson session state and lesson scoring so the selected exercise set is the scored source of truth.

## 1. Approach

Use an audit-first, plugin-local implementation strategy.

Choomfie is a Bun monorepo with a clear runtime split:

- `packages/shared/` owns stable plugin-facing types and interaction registries.
- `packages/core/` owns Discord dispatch, MCP routing, plugin loading, permissions, reminders, and app context.
- `plugins/` owns feature packages loaded through the workspace plugin map in `packages/core/lib/plugins.ts`.

Tutor v3 should keep almost all behavior inside `plugins/tutor/`. That matches the current architecture from `CLAUDE.md`: plugins import `@choomfie/shared`, self-register commands/buttons via shared registries, and avoid depending on core internals beyond the `PluginContext` contract. Core changes should be limited to tests or to genuinely shared contract changes.

Methodology:

1. Reconcile every requested v3 feature against the current implementation before coding.
2. Keep new tutor behavior behind pure helpers where possible, then call those helpers from Discord interaction handlers. This follows the existing `exercise-generator`, `learner-profile`, `lesson-catalog`, and `lesson-buttons` test patterns.
3. Preserve the current interaction split: shared registries in `packages/shared/interactions.ts`, dispatch/error handling in `packages/core/lib/interactions.ts`, tutor command/button registration in `plugins/tutor/lesson-interactions.ts`.
4. Use additive, idempotent SQLite migrations in `plugins/tutor/core/lesson-db.ts`, matching existing `CREATE TABLE IF NOT EXISTS` style.
5. Avoid moving tutor reminders into core unless a new shared scheduler API is deliberately introduced. The current plugin-local timer plus persisted preferences avoids coupling plugins to `AppContext.reminderScheduler`.
6. Make user-selected modes and richer charts alter the actual active exercise list before scoring. A UI-only mode picker would be wrong because `completeLesson()` currently grades against `lesson.exercises.length`.

Recommended implementation order for remaining/spec-hardening work:

1. Harden chart typing with a structured `ChartExercise` shape while preserving existing simple chart exercises.
2. Add optional sequential chart fill state only if the product wants true multi-blank chart completion.
3. Add user-selected exercise modes by making the active session hold the selected exercise list and by updating score denominators.
4. Add focused tests after each change, using pure helpers first and Discord mocks only when unavoidable.
5. Run the focused tests, then `bun test`.

## 2. Files To Modify/Create

### Shared package

- `packages/shared/types.ts`
  - Expected changes: none.
  - Existing pattern: `Plugin`, `ToolDef`, `ToolResult`, `text()`, and `err()` are the plugin boundary. Keep tutor-specific lesson/session types out of shared unless another plugin needs them.

- `packages/shared/plugin-context.ts`
  - Expected changes: none.
  - Existing pattern: `PluginContext` intentionally exposes only `DATA_DIR`, config, optional MCP notifications, optional Discord, and owner ID. Do not add `ReminderScheduler` only for tutor reminders unless multiple plugins need the same abstraction.

- `packages/shared/interactions.ts`
  - Expected changes: none.
  - Existing pattern: register-only shared maps, no dispatch. Tutor should continue using `registerCommand()` and `registerButtonHandler()`.

### Core package

- `packages/core/lib/plugins.ts`
  - Expected changes: none.
  - Existing pattern: explicit workspace package map already includes `tutor: "@choomfie/tutor"`.

- `packages/core/lib/interactions.ts`
  - Expected changes: none unless a new generic interaction contract is needed.
  - Existing pattern: core invokes plugin `onInteraction()` first, then dispatches slash commands, buttons, and modals through shared registries with `safeHandle()`.

- `packages/core/worker.ts`
  - Expected changes: none.
  - Existing pattern: plugin `init()` runs after Discord ready, and plugin `destroy()` is called during worker shutdown. Tutor timers and DB handles should stay cleaned up in `plugins/tutor/index.ts`.

- `packages/core/test/lesson-buttons.test.ts`
  - Extend if chart or mode-picker helpers change button IDs, answer-token storage, or session state.
  - Existing pattern: inspect `ActionRowBuilder.toJSON()` output and assert stable `custom_id` shape without mocking Discord interactions.

- `packages/core/test/exercise-generator.test.ts`
  - Extend for user-selected mode filtering if a new selection helper is added.
  - Existing pattern: pure Bun tests assert invariants, not randomized exact order.

- `packages/core/test/lesson-catalog.test.ts`
  - Extend if `Lesson` gains `contentSets`, structured chart fields, or generated exercise metadata.
  - Existing pattern: validate all Japanese lessons for duplicate IDs, unresolved prerequisites, unit references, empty exercises, empty answers, degenerate button choices, and reachability.

- `packages/core/test/lesson-db-profile.test.ts`
  - Extend if session/mode/reminder persistence changes.
  - Existing pattern: tmpdir-backed real SQLite, no SQLite mocks.

- `packages/core/test/tutor-prompt.test.ts`
  - Keep current coverage for furigana directives and learner profile injection.
  - Extend if mode-specific or chart-specific profile guidance is added.

- `packages/core/test/srs.test.ts`

  - Existing pattern: real `SRSManager` against tmpdir database.

- `packages/core/test/regression/tools-register.test.ts`

### Tutor plugin

- `plugins/tutor/core/lesson-types.ts`
  - Add an explicit `ChartExercise` interface if implementing full A1:
    - `type: "chart"`
    - `grid: (string | null)[][]`
    - `blanks: { row: number; col: number; answer: string }[]`
  - Consider moving `ExerciseMode`, `ContentItem`, and `ContentSet` from `exercise-generator.ts` into this file if lesson definitions need to declare content sets as first-class lesson metadata.
  - Keep `isButtonExercise()` as the single helper used by lesson rendering/tests.

- `plugins/tutor/core/exercise-generator.ts`
  - Keep current recognition, production, matching, and `generateAllExercises()` behavior.
  - If B1 user choice is implemented, add a pure helper such as `selectExercisesForMode(lesson, mode)` or `buildExerciseSet(contentSets, mode)` so tests can avoid Discord interaction objects.
  - Avoid snapshot tests because distractors are randomized with `Math.random()`.

- `plugins/tutor/core/lesson-engine.ts`
  - Update `completeLesson()` only if the active session can contain a generated/selected exercise list different from `lesson.exercises`.
  - A safe design is to add an optional completion input with `totalExercises` and `exerciseResults`, or persist a session-specific exercise count, rather than silently changing `Lesson.exercises`.

- `plugins/tutor/core/lesson-db.ts`
  - Keep additive migrations only.
  - Current tables: `lesson_progress`, `learner_profiles`, `srs_reminder_settings`.
  - Add a session/mode table only if selected exercise sets need to survive worker restart. If sessions remain in memory, handlers must gracefully expire and ask users to run `/lesson`.

- `plugins/tutor/core/learner-profile.ts`

- `plugins/tutor/core/srs.ts`

## 3. Acceptance Criteria

Baseline acceptance for the current Tutor v3 scope:

- [ ] Tutor plugin stays isolated to `plugins/tutor/` except for tests or explicitly shared contracts.
- [ ] Existing lesson flow still works through `/lesson`, lesson buttons, typed production/cloze answers, completion summary, retry, and next-lesson navigation.
- [ ] Button `customId`s remain under Discord's 100-character limit and never include raw answer text.
- [ ] `tutor_prompt` includes active lesson furigana guidance only when a user has an active lesson session for the active module.
- [ ] `tutor_prompt` appends learner profile text when a profile exists and omits it when absent.
- [ ] SRS reminder preferences persist across worker restarts and default to enabled.
- [ ] SRS reminders respect the minimum due threshold and 24-hour cooldown.
- [ ] DM reminder failures do not throw out of the reminder loop.
- [ ] Lesson catalog has no duplicate IDs, unresolved prerequisites, missing unit references, unreachable lessons, empty lessons, empty answers, duplicate button options, self-distractors, or impossible mastery thresholds.
- [ ] `random_word` returns a word from `n5-vocab.json` with spoiler-hidden meaning and remains included in tutor `userTools`.
- [ ] All DB migrations are additive and idempotent.
- [ ] Plugin `destroy()` clears timers and closes Lesson/SRS DB handles.
- [ ] `bun test` passes.

Additional acceptance if full A1 sequential chart completion is implemented:

- [ ] `ChartExercise` has structured `grid` and `blanks` data, not only a rendered prompt string.
- [ ] Chart rendering highlights or identifies the current blank and advances through all blanks in order.
- [ ] Chart answer buttons use short option tokens stored in active session state.
- [ ] Scoring records one result per chart exercise or a deliberately documented per-blank score; the lesson denominator matches that decision.
- [ ] Expired chart sessions respond with a clear ephemeral message telling the user to run `/lesson`.

Additional acceptance if B1 user-selected modes are implemented:

- [ ] Starting a lesson can present a mode picker before exercises.
- [ ] The selected mode determines the actual exercise set stored on the active session.
- [ ] Completion scoring uses the selected exercise set length, not the original `lesson.exercises.length`.
- [ ] Resuming an active lesson preserves the selected mode while the worker is alive.
- [ ] If the worker restarts mid-lesson, the user gets a safe resume/expire path without corrupting progress.
- [ ] Mode selection does not break lessons with hand-written grammar or cloze exercises.

Explicitly deferred unless separately requested:

- Random word daily auto-post scheduling.
- LLM-rewritten learner profiles.
- Persisting full active lesson sessions across worker restarts.
- Moving tutor reminders onto the core reminder scheduler.

## 5. Edge Cases & Risks

- Discord `customId` limit: custom IDs are capped at 100 characters. Keep IDs to action, lesson ID, exercise index, and short token only.
- Answer text separators: answers can contain `:` or long Unicode text. Never parse answers from `customId`.
- Discord button row limits: a row supports up to 5 buttons. Multi-mode or chart options must chunk rows if they exceed this.
- Active sessions are in memory: worker restart loses `activeSessions`. Handlers must expire gracefully instead of saving corrupt progress.
- Score denominator drift: if selected modes or multi-blank charts create a session-specific exercise set, scoring must use that same set.
- Chart scoring ambiguity: decide whether chart blanks count individually or as one exercise before implementing sequential fill.
- Lesson resume ambiguity: persisted `current_exercise` may not be enough to resume selected modes after restart if the generated exercise set was randomized.
- Import-time randomness: lesson helper functions use `Math.random()` for distractors. Tests should assert invariants, not exact order.
- SRS reminder spam: cooldown must be persisted before or immediately after successful DM send. Current implementation records only after send, which avoids recording failed reminders.
- SRS reminder delivery failures: users may block DMs. The reminder loop must continue for other users.
- Plugin boundary creep: adding `ReminderScheduler` or `MemoryStore` to `PluginContext` makes every plugin see core internals. Avoid unless a broader shared API is needed.
- Slash command deployment: plugin commands are registered by importing plugin modules. Verify startup deployment sees tutor commands before relying on manual deploy behavior.
- SQLite migration safety: never rewrite existing progress/profile rows during migration. Use additive columns/tables and sensible defaults.
- Furigana prompt precedence: append lesson-specific directives after generic level guidance so they override broader N5 instructions.
- Timer cleanup: any test or boot path that initializes the tutor plugin must call `destroy()` or otherwise clear timers.
