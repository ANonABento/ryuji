## 1. Approach

### Why the spec needs adjustment, not just blind execution

`scanner-form.tsx` in get-me-job already has:
- Resume upload + paste fallback
- A JD URL scrape input + a `Paste job description (optional)` `<Textarea>`
- A wired call to `scanResume(profile, resumeText, job, fileMeta)` where `job` is built from `textToJob(jobText)` when present
- The downstream `analyzeKeywords()` (in `lib/ats/analyzer.ts`) already produces `KeywordAnalysis[]` with `found: boolean`, synonym matching, and a match rate

So the literal claim "ATS Scanner currently scores resume in isolation" is half-wrong: the JD path exists, but its results are buried inside the existing `ScoreDisplay` / scoring-axes grid and the user gets no foreground "match%" card or missing-keyword chips. The user-visible promise of the spec is the chip UI + dedicated card, not the scoring math (which already exists).

Correct framing: **build a focused presentation layer** for resume-vs-JD comparison, plus a self-contained, testable JD keyword extractor we own (so the ATS analyzer's keyword pipeline can later dedupe against this ŌĆö out of scope for this PR).

### Methodology

1. **Extract** a stand-alone `jd-keywords.ts` module that owns JD-side parsing ŌĆö single-words, multi-word phrases (bigrams/trigrams), simple section weighting (Requirements > Responsibilities > body). It does NOT replace `extractJobKeywords` in `text-to-profile.ts` yet ŌĆö that remains the source of truth for `JobDescription.keywords`. Justification: the spec lists it as new; deduping/replacement can come later once we trust it.
2. **Compute** a separate `match-score.ts` that takes (a) the candidate text from the resume profile and (b) the JD keyword set, returns `{ matchPct, matched, missing }`. Reuse `getSynonyms` from `synonyms.ts` to keep semantic match consistent with the existing analyzer. Pure function, easy to unit-test.
3. **Render** a thin `<JDMatchCard>` that consumes the `match-score` result. Match% number on the left (reuse the radial pattern from `ats-score-card.tsx` or stay flat ŌĆö see decision below), top-10 missing-keyword chips on the right. Click chip ŌåÆ `navigator.clipboard.writeText` + `addToast({ type: "success", title: "Copied: <kw>" })`. Empty state when JD blank.
4. **Wire** the card into `scanner-form.tsx` only when both `result` (post-scan) and `jobText.trim().length >= 20` are present. Place above `FixSuggestionsList` so it sits between the score axes and the suggestions ŌĆö the "what to fix to match this JD" sits adjacent to "what to fix in general".

### Why this split (3 files vs. inlining)

- `jd-keywords.ts` is the only piece with non-trivial token logic worth unit-testing in isolation. Splitting it gives a clean test surface.
- `match-score.ts` becomes trivially testable without rendering React.
- `jd-match-card.tsx` is a presentation component ŌĆö render/snapshot tests via RTL are the right granularity, no logic tests needed.
- This mirrors the existing layout: `analyzer.ts` (logic) + `ats-score-card.tsx` (view).

### Render decision (flat number vs. radial)

Use a **flat % number with a thin progress bar** (the existing `Progress` component in `components/ui/progress.tsx`) instead of another radial. The score-display radial is the hero element; a second radial competes visually. A horizontal bar reads as "match against this JD" (linear, comparative) which is semantically right.

## 2. Files to modify / create

### Create

#### `apps/web/src/lib/ats/jd-keywords.ts`
- `extractJDKeywords(jdText: string, opts?: { max?: number }): JDKeyword[]`
- `interface JDKeyword { term: string; weight: number; source: "requirement" | "body" }`
- Reuse the stop-word set (extract a `STOP_WORDS` const ŌĆö currently duplicated in `analyzer.ts:564-674` and `text-to-profile.ts:295-374`; this module gets its own copy initially, dedupe is **out of scope** to keep the diff small).
- Tokenize ŌåÆ drop stopwords ŌåÆ score by frequency ├Ś section weight (Requirements section = 2x, body = 1x; `extractRequirements` logic in `text-to-profile.ts:390` shows the section header pattern to reuse).
- Detect bigrams/trigrams via sliding window; keep ones whose individual words are non-stopwords AND that occur Ōēź 2 times. Limit to `max ?? 30`.
- Lowercase, trim, dedupe; preserve longest match (if "react native" appears, drop standalone "react" only if "react" isn't independently more frequent ŌĆö keep heuristic simple: include both, let UI dedupe by display).
- Pure function, no React, no DOM. Existing patterns to mirror: `text-to-profile.ts:292-388` (`extractJobKeywords`).

#### `apps/web/src/lib/ats/match-score.ts`
- `computeJDMatch(resumeText: string, jdKeywords: JDKeyword[]): JDMatchResult`
- `interface JDMatchResult { matchPct: number; matched: MatchedKeyword[]; missing: MatchedKeyword[] }`
- `interface MatchedKeyword { term: string; weight: number; matchType?: "exact" | "synonym"; matchedTerm?: string }`
- Mirror the matching idiom in `analyzer.ts:451-507`: word-boundary regex on a `normalizeText(resumeText)`, then `getSynonyms()` fallback with `SYNONYM_MATCH_WEIGHT = 0.8` from `synonyms.ts`. Reuse `normalizeText` ŌĆö extract it from `analyzer.ts:124-130` into this file (do NOT refactor analyzer to import from here in this PR; copy is fine for one helper, refactor is out of scope).
- `matchPct = round(100 * sum(matched weight ├Ś matchTypeWeight) / sum(all keyword weight))`. This weights important JD terms more ŌĆö e.g. matching "react" (in requirements, weight 2) counts more than matching "team" (body, weight 1).
- Sort `missing` by `weight` descending so the chip UI naturally shows highest-impact misses first.
- Pure function. No React.

## 5. Edge cases & risks

1. **Worktree mismatch** (above) ŌĆö top risk. If a worker runs this plan in the Choomfie repo, the file paths won't exist; resolve by re-rooting to get-me-job before any edit.
2. **Spec file missing** ŌĆö `docs/audit/feature-specs/2026-05-10-improve-01-ats-vs-jd.md` is referenced but does not exist in either repo. The brief is the only authority. If a spec is later produced and disagrees with this plan, the spec wins.
3. **Existing JD textarea / `textToJob` already runs** ŌĆö re-implementing JD parsing in `jd-keywords.ts` introduces a second source of truth alongside `extractJobKeywords` in `text-to-profile.ts`. They will produce different keyword sets, and both will appear in the UI (analyzer keywords drive the score breakdown, our keywords drive the new card). This is intentional for this PR (smaller diff, isolated test surface) but the next refactor should converge them ŌĆö flag in the PR description as known follow-up.
4. **`scanResume` already produces `KeywordAnalysis[]`** ŌĆö there's an argument for *not* writing `match-score.ts` and instead deriving the card from `result.legacy.keywords`. Rejected because (a) the brief explicitly lists `match-score.ts` as a new file with tests, (b) `KeywordAnalysis` doesn't carry per-keyword weight, so requirement-section vs body weighting can't be expressed, (c) we'd lose the "JD-side weight" axis that makes the missing chips ranked usefully. Acknowledge in the PR description.
5. **Stopword duplication** ŌĆö three different stopword sets now (`analyzer.ts`, `text-to-profile.ts`, new `jd-keywords.ts`). Flag for a follow-up consolidation; do not refactor in this PR.
6. **Clipboard API** ŌĆö fails silently in HTTP, sandboxed iframes, and some Safari versions. Defensive `if (!navigator.clipboard)` branch with error toast (covered above).
7. **Toast provider scope** ŌĆö `useToast()` throws if not under `<ToastProvider>`. The marketing route group's layout must include the provider. Verify by running the page in dev (`pnpm dev`, port 3000 default) ŌĆö if it throws, escalate (the right fix is upstream layout, out of scope of this card).
8. **Word-boundary on non-ASCII** ŌĆö JS `\b` is ASCII-only, so JD terms with accents (e.g. "r├®sum├®") under-match. Existing analyzer has the same limitation; matching parity is acceptable. Don't try to fix here.
9. **Accessibility** ŌĆö chip `<button>`s need `aria-label="Copy keyword <term>"` (the visible content is "+ <term>", which is fine but screen readers benefit from intent).
10. **Performance** ŌĆö `useMemo` keyed on `[resumeText, jobText, parsedProfile?.rawText]` keeps recomputation off the render path. Big resumes (10k chars) ├Ś big JDs (5k chars) are well under any perceivable threshold for the regex-only pipeline.
11. **Locale support** ŌĆö `page.tsx` accepts `locale`. The card text is currently English-only. Defer i18n strings to whatever pattern the rest of `(marketing)` uses; if `getTranslations` is the convention, drop the strings into the messages JSON; if hardcoded strings are the convention there, match it. Discover by reading sibling files in `(marketing)` before final commit.
12. **Existing `analyzer.ts` keyword score** ŌĆö do not touch. The new card sits next to it, doesn't replace it.
