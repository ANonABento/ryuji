## 1. Approach

Heuristic-only, fully synchronous, client-side. Five metric calculators feed a single `analyzeAnswer(transcript, durationMs)` orchestrator that returns a typed `AnswerFeedback` scorecard. After each `onSubmitAnswer`, the active-session UI renders an `AnswerFeedbackCard`. After the final question, `InterviewSummary` renders a `SessionFeedbackSummary` aggregating all per-answer scorecards.

**Why heuristic, not LLM**:
- Spec explicitly defers LLM feedback (out of scope).
- Zero added latency (network/cost), no streaming UI complexity.
- 70% of coaching value (filler/STAR/quantification/length/pace) is purely lexical — well-suited to deterministic detection that's also unit-testable with fixtures.

**Why client-side**:
- Transcripts already live in client state (`session.answers[i]`); no DB/API round-trip needed for feedback.
- Determinism keeps fixture tests trivial.
- Avoids touching `/api/interview/sessions/[id]/answer` route or DB schema.

**Why per-answer + aggregate (not real-time)**:
- Spec excludes "real-time during-answer indicators."
- Computing once on submit avoids re-renders during typing/voice transcription and keeps state simple.

**Duration source**:
- For voice mode, `useInterviewSession` does not currently capture answer duration. We need an answer-start timestamp.
- Cleanest approach: track `answerStartedAt: number | null` inside `InterviewActiveSession` (set to `Date.now()` when `currentQuestion` changes; capture `Date.now() - answerStartedAt` at submit time and pass it to `analyzeAnswer`). This avoids changing `InterviewSession` shape or the persistence layer.
- Fallback when duration is unknown (e.g. resumed past session): mark `durationSec`/`wordsPerMinute` as `null` and skip those two metric tiles in the card.

### New — `src/lib/interview/`

Pattern reference: `src/lib/interview/format-for-docs.ts` and its test (pure-function lib + colocated `*.test.ts` using vitest).

1. **`src/lib/interview/filler-words.ts`** — `countFillers(transcript: string): { total: number; byWord: Record<string, number> }`.
   - Built-in list: `um, uh, er, ah, like, you know, i mean, basically, literally, actually, sort of, kind of, right, so` (configurable constant `FILLER_WORDS`).
   - Word-boundary regex per filler; `you know` / `i mean` / `sort of` / `kind of` matched as bigrams. Case-insensitive. Don't match inside words (`like` ≠ `unlikely`).
   - Returns counts both summed and per-word so the UI can show "you said 'um' 8 times".

2. **`src/lib/interview/star-detector.ts`** — `detectSTAR(transcript: string): { situation: number; task: number; action: number; result: number }` (each 0–1).
   - Keyword buckets per dimension:
     - **Situation**: `at <company>`, `when i was`, `back when`, `the context`, `the company`, `our team was`, `the situation`.
     - **Task**: `i was responsible for`, `my goal`, `i needed to`, `i had to`, `was tasked with`, `the challenge was`, `my role`.
     - **Action**: `i designed`, `i built`, `i implemented`, `i led`, `i decided`, `i proposed`, `i refactored`, `i wrote`, `i coordinated`, plus generic `i <verb>` heuristic where verb ∈ action-verb list.
     - **Result**: presence of metrics (delegate to `extractMetrics`) **or** keywords `as a result`, `we shipped`, `which led to`, `improved`, `reduced`, `increased`, `saved`, `grew`.
   - Score: `min(1, hits/2)` per bucket — diminishing returns (one match = 0.5, two = 1.0). Tunable.

3. **`src/lib/interview/feedback.ts`** — orchestrator + types + extractors.
   ```ts
   export interface AnswerFeedback {
     fillers: { total: number; byWord: Record<string, number> };
     star: { situation: number; task: number; action: number; result: number };
     quantificationCount: number;
     durationSec: number | null;
     wordsPerMinute: number | null;
     wordCount: number;
     targetDurationSec: { min: number; max: number };
     suggestion: string;
     ratings: {
       fillers: "green" | "yellow" | "red";
       star: "green" | "yellow" | "red";
       quantification: "green" | "yellow" | "red";
       length: "green" | "yellow" | "red" | "unknown";
       pace: "green" | "yellow" | "red" | "unknown";
     };
   }

   export function analyzeAnswer(
     transcript: string,
     options: { durationMs: number | null; category: InterviewQuestionCategory },
   ): AnswerFeedback;

   export function extractMetrics(transcript: string): string[];
   export function deriveTopSuggestion(scorecard: Omit<AnswerFeedback, "suggestion">): string;
   export function aggregateFeedback(scorecards: AnswerFeedback[]): SessionFeedback;
   ```
   - **Quantification regex**: `/\b\d+(?:\.\d+)?\s*(?:%|percent|x|hours?|days?|weeks?|months?|years?|users?|customers?|requests?|ms|seconds?|k|m|million|thousand)?\b/gi` plus currency `/\$\d/`.
   - **Thresholds (rationale: spec values + common interview-coaching guidance)**:
     - Fillers: green ≤ 3, yellow 4–8, red > 8 (per answer).
     - Length: target 60–90s (behavioral / situational / cultural-fit / general) or 90–180s (technical). Green inside ±10% window, yellow ±25%, red beyond.
     - Pace: green 130–150 wpm, yellow 110–129 or 151–170, red outside.
     - Quantification: green ≥ 2, yellow 1, red 0 (mark "skip" on culture-fit if needed — keep simple, use red).
     - STAR: green if all four ≥ 0.5; yellow if any of {situation, action} ≥ 0.5 but result < 0.5; red otherwise. Tuned for behavioral; for technical, weight {task, action} only.
   - **`deriveTopSuggestion`** priority order (return first applicable):
     1. STAR result missing → "Add a quantified result — what changed because of your action?"
     2. Quantification = 0 → "Mention a number — duration, scale, %, dollars, or count."
     3. Fillers > 8 → "Cut filler words — try a deliberate pause instead of 'um' / 'like'."
     4. Length far below target → "Go deeper — this answer was X seconds, target is N–M."
     5. Length far above target → "Trim it down — this answer was X seconds, target is N–M."
     6. Pace too fast → "Slow down — you spoke at X wpm, aim for 130–150."
     7. Pace too slow → "Pick up the pace a touch — X wpm, aim for 130–150."
     8. Fallback: "Solid answer. Tighten one detail next round."
   - **`aggregateFeedback`** averages each metric across answered (non-skipped, non-empty) answers; surfaces the most common rating per metric and the most-frequently-suggested top-suggestion bucket.

## 5. Edge cases & risks

- **Duration is unknown** (text mode, no recording): UI must degrade. Show "—" for length/pace. `aggregateFeedback` should ignore unknown values when computing averages, not zero them.
- **Skipped answers** (`"[skipped]"`) and empty strings: excluded from per-answer feedback display and from the aggregate.
- **Voice transcripts contain no punctuation**: word-boundary regex handles this; sentence-based heuristics (none here) would not. Confirm test fixtures include a punctuation-free fixture.
- **Filler-word false positives**: `"like"` is heavily used legitimately ("things like X, Y, Z"). Heuristic over-counts. Acceptable for v1 — note in PR description, document the known false positives in a brief code comment in `filler-words.ts`.
- **STAR detector is genuinely fuzzy**. Real answers don't always say "I was tasked with". Scoring is intentionally generous (0.5 per single hit) to avoid screaming "missing TASK!" at every reasonable answer. Tune via the three test fixtures.
- **Pace calc**: `(words / durationMs) * 60_000` only sensible if both > 0. Guard `durationMs > 0`.
- **Performance Insights box duplication** (`interview-summary.tsx`): existing block already shows "Questions Answered / Response Quality / Feedback Received". Decision: **keep both, place `<SessionFeedbackSummary>` above the existing block.** They measure different things (existing is qualitative-LLM-based; new is heuristic-quant). Avoids removing existing copy users may have shared/screenshotted.
- **Re-render churn**: scorecards held in a ref map keyed by question index. Only `lastSubmittedFeedback` is component state (cleared on advance). No infinite-render risk because `analyzeAnswer` is pure and called once per submit.
- **i18n**: codebase has `messages/` and `translate-messages.ts`. Filler-word list is English-only. Acceptable for v1 — add a `// TODO: i18n filler list` comment in `filler-words.ts`.
- **`forbidden-color-lint.cjs`**: enforces semantic tokens (`success` / `warning` / `destructive`). Use those, not raw hex/Tailwind palette colors.
- **`forbidden-time-lint.cjs`**: likely forbids `new Date().toISOString()` / `Date.now()` outside approved helpers. Verify by reading the lint script before using `Date.now()` in `InterviewActiveSession`. If forbidden, use whatever `@/lib/format/time` (or equivalent) helper is already approved (`nowIso()` or similar is already used by `format-for-docs.ts`).
- **Bento-ya routing bug**: this task arrived in the wrong worktree. Surface to the operator; do not silently implement in the Choomfie repo (which has no `apps/web/`).
