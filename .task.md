# Tutor v3 Implementation Plan

Spec: `docs/tutor-v3-spec.md`

This plan is based on the current checkout, not a greenfield reading of the spec. The tree already contains a substantial Tutor v3 implementation under `plugins/tutor/` plus Bun tests under `packages/core/test/`; `bun test` currently passes with 66 tests across 13 files.

## Current State Audit

Implemented or mostly implemented in the current tree:

- A1 chart review exists through `plugins/tutor/modules/japanese/lessons/kana-helpers.ts` and `Exercise.type === "chart"` in `plugins/tutor/core/lesson-types.ts`; it is represented as a button exercise whose prompt contains a rendered grid.
- A2 `random_word` exists in `plugins/tutor/tools/random-word.ts`, is aggregated by `plugins/tutor/tools/index.ts`, and is exposed in `plugins/tutor/index.ts` `userTools`.
- A3 due-card reminders exist in `plugins/tutor/index.ts` via a worker-local timer and `SRSManager.getDueCountByUser()` in `plugins/tutor/core/srs.ts`.
- A4 `furiganaLevel` exists on `Lesson`, flows through `TutorPromptContext`, and is passed into `japaneseModule.buildTutorPrompt()` by `plugins/tutor/tools/tutor-tools.ts`.
- A5 authoring guidance exists in `plugins/tutor/LESSON_AUTHORING.md`.
- B1 content-set generation exists in `plugins/tutor/core/exercise-generator.ts`; matching exercises are handled as button exercises.
- B2 deterministic learner profiles exist in `plugins/tutor/core/learner-profile.ts` and `LessonDB` `learner_profiles`.
- B3 units 2-4 exist and are registered by `plugins/tutor/modules/japanese/lessons/index.ts`.
- Tests already cover SRS basics, exercise generation, learner profile updates, learner profile DB round trips, lesson catalog shape, plugin loading, and tool counts.

Remaining spec gaps and hardening work before considering Tutor v3 complete:

- B1 user-selected mode picker is not implemented. Current lessons contain concrete exercises, so selecting a subset at runtime would need careful scoring semantics.
- A3 opt-out/persistence is not implemented. The 24h reminder cooldown is in memory and resets on worker restart.
- A1 is a pragmatic one-blank chart exercise, not the richer structured multi-blank `ChartExercise` model described in the spec.
- Button `customId`s now use short session-scoped option tokens, with the full answer text kept in the active session for scoring.
- `tutor_prompt` is covered for module-level furigana directives, active lesson furigana context, and learner profile injection.

## 1. Approach

Use an audit-first, incremental completion strategy. The repo already follows a plugin-centered architecture: plugin code imports only `@choomfie/shared` types and shared interaction registries, while core owns Discord dispatch, MCP routing, and plugin loading. The plan preserves that boundary instead of moving tutor behavior into `packages/core`.

Methodology:

1. Reconcile the implemented surface against `docs/tutor-v3-spec.md`.
2. Keep runtime changes inside `plugins/tutor/` unless shared typing or tests require otherwise.
3. Prefer data-driven lesson definitions and pure helper functions over Discord stubs; this matches the existing `packages/core/test/` style.
4. Use idempotent SQLite DDL in `LessonDB` for profile/reminder metadata, matching the existing `CREATE TABLE IF NOT EXISTS` pattern.
5. Treat the user mode picker as a deliberate design choice, not a quick UI toggle, because lesson completion currently scores against the registered concrete `lesson.exercises` list.

Scope decision:

- Ship the current v3 core as the baseline: chart practice, random word tool, SRS due reminders, furigana-aware tutor prompts, deterministic learner profile, generated exercises, and units 2-4.
- Add hardening where the current implementation can fail at runtime: answer custom IDs, prompt tests, persistent/opt-out reminder behavior if the product decision is to keep reminders enabled by default.
- Defer random-word auto-posting because the spec marks it optional.
- Defer B1 user-selected mode picker unless the implementation also changes lesson progress scoring to record the selected exercise set. A cosmetic picker that still grades against the full lesson would be incorrect.

## 5. Edge Cases & Risks

- Discord button `customId` limit: raw answers can exceed 100 characters or contain separators. Use stable option indices and keep the answer text in session memory.
- Session lifetime: active lesson sessions are in memory and disappear on worker restart. Handlers must gracefully tell the user to restart `/lesson`.
- Mode picker scoring: selecting only recognition or matching exercises cannot use the current `completeLesson()` count without changing the score denominator. This is the main reason to defer B1 user choice unless scoring is redesigned.
- Import-time randomness: lesson files and generators use `Math.random()` for distractors. Tests must assert invariants, not order.
- SRS reminder spam after restart: the current in-memory cooldown map resets. Persist cooldown if reminders remain enabled by default.
- SRS reminder delivery failures: users may block DMs. Failures should be swallowed without logging noisy stack traces.
- Plugin boundary: exposing core reminder scheduler through `PluginContext` would couple plugins to core internals. Prefer a plugin-local reminder table plus direct DM, or make a deliberate shared API if reuse is required.
- Slash command deployment: startup auto-deploy should see plugin commands after plugins load, but the standalone deploy script may not include plugin commands. Verify before relying on manual deploy for `/lesson` and `/progress`.
- SQLite migration safety: only additive, idempotent DDL should be used. Do not rewrite existing lesson/profile rows during migration.
- Furigana prompt precedence: append lesson-specific furigana guidance after the level guide so it overrides broad N5 defaults without editing every guide.
- Timer cleanup in tests and worker restarts: any code path that calls plugin `init()` must call `destroy()` or clear timers.
