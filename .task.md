# Tutor v3 Implementation Plan

Task description: `docs/tutor-v3-spec.md`

This plan is based on the current checkout and on the architecture notes in `CLAUDE.md`. The repository is not greenfield: much of Tutor v3 already exists under `plugins/tutor/`, with regression coverage in `packages/core/test/`. The implementation work should therefore be a completion and hardening pass, not a broad rewrite.

## Current State Audit

Already present in the current codebase:

- Plugin boundary: plugins import `Plugin`, `ToolDef`, `PluginContext`, and interaction registries from `@choomfie/shared`; core owns dispatch, MCP routing, config, Discord, and plugin loading.
- Tutor plugin export pattern in `plugins/tutor/index.ts` matches `plugins/browser/index.ts`, `plugins/voice/index.ts`, and `plugins/socials/index.ts`: a single `Plugin` object, module-level runtime state where needed, `init()` for startup work, and `destroy()` for cleanup.
- A1 chart practice exists through `chartReview()` and `renderChartGrid()` in `plugins/tutor/modules/japanese/lessons/kana-helpers.ts`, with chart exercises included in hiragana and katakana lessons. It is currently a one-blank exercise encoded as the generic `Exercise` shape, not the richer structured multi-blank `ChartExercise` from the spec.
- A2 `random_word` exists in `plugins/tutor/tools/random-word.ts`, is aggregated by `plugins/tutor/tools/index.ts`, exposed by `plugins/tutor/index.ts`, and covered by plugin tool registration tests.
- A3 SRS due reminders exist in `plugins/tutor/index.ts` with a 4-hour timer, a 5-card threshold, DM delivery, and persisted opt-out/cooldown state in `LessonDB`. The implementation uses plugin-local DM sending instead of core `ReminderScheduler`, which avoids widening `PluginContext`.
- A4 `furiganaLevel` exists on `Lesson`, is read from the active lesson session by `tutor_prompt`, and is honored by `japaneseModule.buildTutorPrompt()`.
- A5 vocab-first authoring guidance exists in `plugins/tutor/LESSON_AUTHORING.md`, and lesson prerequisites enforce the ordering.
- B1 content-set generation exists in `plugins/tutor/core/exercise-generator.ts` with recognition, production, matching, and `generateAllExercises()`. User-selected mode picking before a lesson is not implemented.
- B2 deterministic learner profiles exist in `plugins/tutor/core/learner-profile.ts`, are stored in `LessonDB.learner_profiles`, and are appended to `tutor_prompt` output.
- B3 units 2-4 exist and are registered through `plugins/tutor/modules/japanese/lessons/index.ts`: 8 katakana lessons, 8 phrase lessons, and 6 grammar lessons.
- Tests already cover exercise generation, learner profile logic and DB round trips, tutor prompt context, button custom IDs, lesson catalog shape, SRS basics, plugin loading, tool registration, and boot smoke behavior.

Primary remaining product gaps relative to the spec:

- Structured chart exercises should be made explicit if the target is exact A1 compliance: `ChartExercise` data, multiple blanks, current blank state, and chart-specific rendering/scoring.
- User-selected practice mode picker should be added only if lesson scoring records the exact active exercise set. A picker that still grades against `lesson.exercises.length` would be wrong.
- Random word daily auto-posting remains optional in the spec and should stay deferred until a channel/config story exists.
- Unit 2 is intentionally 8 lessons in the current tree, while the draft spec sketches 10. Decide whether this is acceptable before adding lesson content.

## 1. Approach

Use an audit-first, incremental completion strategy.

The methodology:

1. Treat `docs/tutor-v3-spec.md` as the product task and the current code as the implementation baseline.
2. Keep feature code inside `plugins/tutor/` unless a change is truly shared infrastructure. This preserves the monorepo pattern where plugins depend on `@choomfie/shared`, while `packages/core` owns Discord dispatch, MCP tools, app context, config, plugin loading, and interaction routing.
3. Prefer pure helpers plus focused Bun tests for lesson generation, scoring, profile updates, and button construction. Existing tests in `packages/core/test/` import plugin modules directly instead of mocking all of Discord.
4. Use additive, idempotent SQLite migrations in `LessonDB`, matching existing `CREATE TABLE IF NOT EXISTS` and `INSERT ... ON CONFLICT` patterns.
5. Avoid widening `PluginContext` unless several plugins need the same new capability. The current direct-DM SRS reminder path is acceptable because it keeps the reminder behavior plugin-local and avoids coupling Tutor to core `ReminderScheduler` internals.
6. Make UI/session changes by first separating pure state transitions from Discord interaction plumbing. This keeps the 3-second Discord interaction constraint easy to satisfy and keeps tests small.
7. Validate through the existing regression suite first, then add only targeted tests for new behavior.

Recommended implementation order:

1. Baseline verification: run focused tests and confirm current Tutor v3 behavior before changing logic.
2. Structured chart hardening: add explicit chart data types and helper functions, then adapt current chart exercises without changing lesson progression semantics.
3. Optional mode picker: only implement if the product decision is to let users choose recognition, production, or matching at lesson start. Store the generated exercise list on `ActiveLessonSession` and score against that list.
4. Content alignment: decide whether Unit 2 must match the draft's 10-lesson outline or whether the current 8-lesson condensed unit is accepted.
5. Documentation cleanup: update `LESSON_AUTHORING.md` and this plan if the richer chart or mode-picker behavior changes lesson authoring conventions.

Explicit deferrals unless separately requested:

- Random-word daily auto-post scheduling.
- LLM-rewritten learner profiles.
- Moving SRS reminders onto core `ReminderScheduler`.
- New external dependencies.

## 2. Files To Modify Or Create

### Shared package

- `packages/shared/types.ts`
  - Keep this as the stable plugin contract: `Plugin`, `ToolDef`, `ToolResult`, `text()`, and `err()`.
  - No expected change for chart hardening or mode picker.
  - Only modify if the final design introduces a generally useful shared plugin capability.

- `packages/shared/plugin-context.ts`
  - No expected change for the preferred plan.
  - Do not add core-only services such as `ReminderScheduler` for Tutor alone. If reminders are later generalized across plugins, add a narrow shared interface here and update core `AppContext` accordingly.

- `packages/shared/interactions.ts`
  - No expected change. Tutor commands and buttons should keep registering through `registerCommand()` and `registerButtonHandler()`.
  - Existing split is important: shared owns registries, core owns dispatch in `packages/core/lib/interactions.ts`.

### Core package

- `packages/core/lib/plugins.ts`
  - No expected runtime change. Tutor is already in the explicit workspace package map with voice, browser, and socials.
  - Use existing plugin loading tests if any tool or plugin export changes.

- `packages/core/lib/interactions.ts`
  - No expected routing change. New Tutor buttons should continue to use the `lesson:*` custom ID prefix and the shared button registry.
  - Keep error handling inside the existing `safeHandle()` wrapper.

- `packages/core/lib/commands.ts`
  - No expected change for Tutor plugin commands. Core slash commands remain core-owned.

- `packages/core/scripts/deploy-commands.ts`
  - Optional verification target. If manual command deployment does not include plugin-registered `/lesson` and `/progress`, document that startup auto-deploy is the supported path or adjust the script to load enabled plugins before reading command definitions.

- `packages/core/test/lesson-buttons.test.ts`
  - Extend for structured chart options and/or mode-picker custom IDs.
  - Preserve the current pattern: test helper output and serialized Discord components, not full Discord interaction mocks.

- `packages/core/test/lesson-catalog.test.ts`
  - Extend catalog invariants if `ChartExercise` is added: chart blanks must resolve to valid grid coordinates, blank answers must be present in options, and chart exercises must not exceed Discord button limits.

### Tutor plugin

- `plugins/tutor/core/lesson-types.ts`
  - If pursuing exact A1 compliance, replace the generic `chart` encoding with a discriminated chart shape:
    - `type ChartBlank = { row: number; col: number; answer: string }`
    - `interface ChartExercise extends BaseExercise { type: "chart"; grid: (string | null)[][]; blanks: ChartBlank[]; ... }`
  - Consider splitting the current broad `Exercise` interface into a union only if it reduces complexity. Keep the edit conservative; many existing lessons already depend on the current shape.
  - If mode picker is implemented, move `ContentItem`, `ContentSet`, and `ExerciseMode` here from `exercise-generator.ts` only if multiple modules need those types. Otherwise leave them local to the generator.

- `plugins/tutor/core/exercise-generator.ts`
  - Keep recognition, production, matching, and `generateAllExercises()`.
  - For mode picker, add a pure selection API such as `selectExercisesForMode(content, mode)` or `resolveLessonExercises(lesson, selectedMode)`.
  - Make scoring tests possible without Discord by returning a concrete exercise array for a selected mode.

- `plugins/tutor/core/lesson-engine.ts`
  - If mode picker is implemented, update completion logic so the score denominator is the active session's selected/generated exercises, not always the persisted `lesson.exercises`.
  - Keep mastery threshold behavior and prerequisite unlocking consistent with existing `startLesson()`, `scoreExercise()`, and `completeLesson()` patterns.

- `plugins/tutor/core/lesson-db.ts`
  - Keep additive migrations only.
  - Existing `learner_profiles` and `srs_reminder_settings` tables should remain the source of truth for profile and reminder state.
  - Add storage only for preferences that must survive worker restart. Do not store transient active lesson sessions unless restart resume becomes a product requirement.

- `plugins/tutor/core/learner-profile.ts`
  - Keep deterministic updates for lesson completions and SRS reviews.
  - If mode picker is implemented, update preferred exercise type based on the selected mode and observed scores.

- `plugins/tutor/core/srs.ts`
  - Keep SRS scheduling and due-count queries inside `SRSManager`.
  - Do not move Discord sending into SRS. `SRSManager` should remain storage and scheduling logic, while `plugins/tutor/index.ts` owns plugin runtime behavior.

- `plugins/tutor/index.ts`
  - Preserve initialization order: create SRS DB, create Lesson DB, register lessons, initialize modules, then start reminder timers.

- `plugins/tutor/lesson-interactions.ts`

## 3. Acceptance Criteria

- [ ] Tutor plugin still exports a valid `Plugin` object with stable `tools`, `instructions`, `userTools`, `init()`, and `destroy()` fields.
- [ ] All enabled Tutor tools remain available through `getAllTutorTools()` and the plugin `tools` array.
- [ ] `/lesson` and `/progress` remain registered through shared command registries and dispatch through core interaction routing.
- [ ] Lesson button custom IDs never include raw answer text and stay within Discord's 100-character custom ID limit.
- [ ] Long button labels are truncated to Discord-safe lengths while full answers remain available for scoring.
- [ ] Chart exercises render a readable grid and score the intended blank answer.
- [ ] If structured charts are implemented, every chart blank references an in-bounds grid cell and has a non-empty answer.
- [ ] If multi-blank charts are implemented, user progress advances through blanks deterministically and stale buttons are rejected gracefully.
- [ ] If a mode picker is implemented, the chosen mode determines the concrete active exercise list before scoring starts.
- [ ] If a mode picker is implemented, lesson completion score uses the active exercise list denominator, not the original lesson definition by accident.
- [ ] Typed production and cloze answers still work through the plugin `onMessage()` path.
- [ ] `tutor_prompt` preserves default N5 behavior when there is no active lesson session.
- [ ] `tutor_prompt` includes full, partial, or no-furigana directives when the active lesson sets `furiganaLevel`.
- [ ] `tutor_prompt` appends learner profile text when a profile exists and omits it when no profile exists.
- [ ] Learner profiles update deterministically after lesson completion and SRS review.
- [ ] SRS reminders respect the due-card threshold, persisted opt-out setting, and persisted 24-hour cooldown.
- [ ] SRS reminder timers are not started without Discord and are cleared in `destroy()`.
- [ ] Failed DM delivery for SRS reminders does not throw or spam logs.
- [ ] Lesson catalog has no duplicate lesson IDs, unresolved prerequisites, missing unit references, unreachable lessons, empty lessons, empty answers, duplicate button options, or self-distractors.
- [ ] Unit count/content decision is explicit: either accept the current 8-lesson Unit 2 or expand it to the draft 10-lesson shape.
- [ ] No new runtime dependency is added unless the implementation cannot meet the acceptance criteria without it.
- [ ] `bun test` passes.
- [ ] Manual Discord smoke test passes for `/lesson`, answer flow, lesson completion, `/progress`, `srs_review`, `srs_rate`, `srs_reminders`, `random_word`, and `tutor_prompt` during a lesson.

## 5. Edge Cases And Risks

- Discord custom ID limit: raw answers can exceed 100 characters or contain `:`. Keep option tokens in custom IDs and store answer text in active session state.
- Discord button row limits: buttons are constrained by Discord component limits. Chart and matching options must stay within those limits or paginate/split deliberately.
- Session lifetime: active lesson sessions are in memory and disappear on worker restart. Stale buttons should tell the user to restart `/lesson`.
- Scoring denominator: mode-selected or generated exercises must score against the active exercise list, otherwise completion percentages will be wrong.
- Chart state drift: multi-blank chart exercises can desynchronize if users click old buttons. Validate lesson ID, exercise index, blank index, and option token before scoring.
- Typed answer collisions: production and cloze exercises rely on `onMessage()`. Starting a new lesson while a typing exercise is active can confuse answer routing unless active session replacement stays explicit.
- Import-time randomness: lessons and generators use `Math.random()` for distractors. Tests should assert invariants, not exact ordering.
- Unit content drift: the spec says Unit 2 has 10 lessons; the current implementation has 8. Expanding content may affect progression tests, lesson counts, and user expectations.
- Reminder spam after restart: cooldown must remain persisted in `LessonDB`, not only module-local memory.
- Reminder privacy/delivery: DMs can fail when a user blocks DMs. Failure should be non-fatal and quiet.
- Plugin boundary creep: exposing core services to `PluginContext` for a single feature increases coupling across the monorepo.
- Slash command deployment: startup auto-deploy may see plugin commands after plugins load, while standalone deploy behavior should be verified before relying on it for `/lesson` and `/progress`.
- SQLite migrations: only additive, idempotent DDL is safe for existing user data.
- Prompt precedence: active-lesson furigana guidance must be appended after generic level guidance.
- Worker restart cleanup: timer and manager state must be cleared in plugin `destroy()` to match the hot-reload boundary in `CLAUDE.md`.
