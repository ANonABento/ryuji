## 1. Approach

**Methodology:** regex-based pattern detection over Gmail messages, run from a Vercel cron route, with strict status-advancement rules and an opt-in user toggle.

**Why this shape:**
- Spec explicitly defers LLM classification ("regex patterns are 80% accurate"); start dumb, measure precision, escalate later if needed.
- Cron-driven (not webhook) because Gmail push notifications require Pub/Sub + domain verification ‚Äî out of scope for v1.
- Daily 9 AM UTC matches existing cadence (`follow-ups` already runs at `0 9 * * *` per `apps/web/vercel.json`); reuse `requireCronAuth` and the established `GET ‚Üí JSON` route shape from `apps/web/src/app/api/cron/digest/daily/route.ts`.
- Status only advances along the pipeline (saved ‚Üí applied ‚Üí interviewing ‚Üí offer/rejected). Advancement-only is the user-trust contract: a regex false-positive that *adds* an interview stage is recoverable via Undo; one that *demotes* an offer back to applied is destructive.
- Per-user `last_scanned_at` watermark + paginated Gmail `q=after:` query keeps cron under Vercel's 60s/300s budget even on heavy inboxes.
- Notification + Undo button (existing notifications system in `apps/web/src/lib/db/notifications.ts`) closes the loop: low-friction recovery turns regex precision from a blocking concern into a UX one.

**Status state machine** (advance-only):
```
saved (rank 0) ‚Üí applied (1) ‚Üí interviewing (2) ‚Üí offer (3)
                                                ‚Üò rejected (terminal, allowed from 1+)
```
Implementation: numeric rank table; only update if `newRank > currentRank` (with `rejected` allowed from any non-terminal). Closed sub-statuses (`rejected`, `expired`, `dismissed` per `types/opportunity.ts:48-52`) are terminal ‚Äî never auto-overwrite a closed opportunity.

## 5. Edge cases & risks

- **OAuth token expired mid-run** ‚Äî `gmail.ts` should refresh; if it throws, catch per-user, increment `errors`, continue loop. Don't fail the whole cron on one bad token.
- **Email body encoding** ‚Äî Gmail returns base64url MIME parts. `lib/google/gmail.ts` already handles this (`extractBody` at top of file). Do not reimplement.
- **HTML emails** ‚Äî strip tags before regex; otherwise `<span>Schedule</span>` matches `Schedule` but `interviewing` patterns fail on multi-tag splits. Use a minimal stripper, not a full HTML parser.
- **Marketing/spam false positives** ‚Äî "We extend an offer of 20% off!" matches the offer pattern. Mitigation: require company-name match against an existing opportunity *and* sender domain not in a blocklist (Mailchimp/Constant Contact mailers). Add `BLOCKED_SENDER_DOMAINS` const.
- **Multiple opps for same company** ‚Äî user applied to Anthropic twice. Match the most recently-modified non-closed one. If two are at the same status, log and skip (don't guess).
- **Bulk historical run on first opt-in** ‚Äî 30-day window can return hundreds of messages. Cap per-user to 200 messages per run; if more, process newest first and let watermark advance partially. Spec doesn't address this; flag in PR.
- **Vercel function timeout** ‚Äî default 10s, can extend to 60s/300s on Pro. Loop is per-user; if N users √ó M emails > budget, time-slice (process N/k users per run, rotate). Likely not needed at current scale; add a `MAX_USERS_PER_RUN = 100` guard regardless.
- **Watermark race** ‚Äî if two cron invocations overlap (Vercel retry on 5xx), both could advance the watermark and skip emails between them. Mitigate: take watermark *before* querying, write *after* successful processing, AND wrap status updates in a transaction so partial failure rolls the watermark back.
- **PII / privacy** ‚Äî opt-in default OFF satisfies spec, but document in settings copy that email subjects/snippets are read but never stored verbatim. Persist only `gmailMessageId` (a pointer, not content).
- **Notification fatigue** ‚Äî daily run on a noisy inbox could spawn many notifications. Batch into a single "‚úì Auto-updated 3 opportunities" notification when ‚â•2 advancements per user per run.
- **`rejected` from `offer` regression risk** ‚Äî covered above; chose to skip rather than apply.
- **Pattern collisions** ‚Äî "Thank you for applying" appears in some interview-invite emails too. Order patterns by priority: offer > rejected > interviewing > applied. First match wins. Tested via fixture.
