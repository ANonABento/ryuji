# Audit notes - 2026-03-25

## Scope

Repository audit focused on correctness issues, code-quality cleanup, and documentation drift in the Bun-based Discord/MCP app.

## Fixed in this pass

1. Plugin tool collision handling in `lib/plugins.ts`
The loader could still register a plugin after detecting a conflicting tool name. It now validates the full plugin first and skips the plugin on any duplicate or collision.

2. Reminder scheduling beyond the JavaScript timer ceiling in `lib/reminders.ts`
Long-horizon reminders now use chunked timeout scheduling instead of relying on a single `setTimeout`.

3. Reminder scheduler replacement race during startup in `lib/discord.ts`
Discord startup now initializes the existing scheduler instead of replacing it, preserving reminders scheduled before login completes.

4. Nested config defaults in `lib/config.ts`
Config loading now deep-merges `personas` and `voice` so partial config files do not erase nested defaults.

5. Attachment file handling in `lib/discord.ts`
Attachment downloads now check for non-OK HTTP responses, sanitize filenames, and use unique names to reduce collision and path issues.

6. PID guard safety in `server.ts`
Startup no longer blindly terminates the PID from the pidfile. It first checks whether the target process looks like a Choomfie-related process.

7. README structure drift in `README.md`
The documented project structure now includes the top-level plugin packages.

8. Voice and socials plugin robustness
The voice plugin now clears its module-level manager cleanly and destroys half-open connections on join failure. Social provider fallback now skips invalid configured providers instead of failing immediately, and the `yt-dlp` transcript flow now cleans up temporary subtitle files.

9. Language-learning SRS correctness
The language plugin now enforces card ownership in `srs_rate`, uses SQLite-compatible datetime formatting for SRS due comparisons, and computes the "learned" count against the real empty-card state instead of incorrectly counting untouched cards as learned.

10. Installer hardening
`install.sh` now checks for `curl`, applies restrictive permissions to generated secret files, and avoids appending duplicate PATH exports on repeated installs.

## Remaining risks

1. Voice ingestion path in `plugins/voice/manager.ts`
The current implementation passes Discord-received Opus chunks directly into `ffmpeg -f opus`. This may be brittle depending on framing/container expectations and should be validated against real Discord voice traffic.

2. Tool access control is still not strongly server-enforced
The owner-vs-user restrictions are described in prompts and docs, but there is still no robust server-side authorization layer that binds a specific Discord user context to each MCP tool call. This is the biggest remaining architectural risk.

Follow-up note:
This PR does not attempt to redesign the authorization model. Server-side per-request tool authorization should be handled as a separate change so it can be reviewed as a security-sensitive update.

3. Limited automated coverage outside the newly added tests
There is still no automated regression coverage for reminders, attachment ingestion, Discord tool behavior, or real provider integration failures.

4. Social and plugin provider surfaces
The YouTube/Reddit provider wrappers and fallback behavior were reviewed, but not exercised with live provider failures in this pass.
